# i18n Review Rules

Apply when a change touches shipped user-facing UI text, translation dictionaries, translation helper usage, notification text, translated attributes, i18n lint configuration, or locale formatting.

## Review focus

- Use `.i18n-lint.toml` as the source of truth for i18n lint scope, covered UI helpers, translated attributes, exclusions, and allowlists.
- Inside that scope, all shipped user-facing UI text must use helpers from `frontend.context.i18n`.
- Console text is exempt from i18n.
- Keep out-of-scope developer-only `(Dev)` labels inline in code/config, not in translation dictionaries.
- If a new user-facing surface is not represented in `.i18n-lint.toml`, flag the missing lint coverage.
- Reuse existing `src/resources/dicts/en.edn` keys only on exact semantic owner and textual role match.
- If an existing key does not match exact semantic owner and textual role, follow `docs/i18n-key-naming.md` instead of reusing it.
- Add new English source text to `src/resources/dicts/en.edn`.
- Add non-English entries only when providing actual translations.
- When renaming or removing keys, clean up stale keys in affected locale files.
- `notification/show!` and translated attributes from `.i18n-lint.toml` must not receive raw English string literals unless proven non-user-facing.
- Translated attributes include placeholder, title, aria, and label-like UI text covered by `.i18n-lint.toml`.
- For plain dynamic text, use placeholders like `{1}` and pre-format arguments in the caller before passing them to `t`.
- Keep complete sentences in one translation entry.
- Use `interpolate-rich-text`, `interpolate-sentence`, `locale-join-rich-text`, and `locale-format-*` from `frontend.context.i18n` instead of assembling text ad hoc in the caller.
- Function-valued translations are allowed only for real logic or hiccup rich text.
- Function-valued translations may only use `str`, `when`, `if`, and `=`.
- Rich text and inline links must stay in a single translation entry, not split across multiple keys.
- Preserve emoji/icon glyphs from `en.edn`.
- Use punctuation natural to each locale.
- Pluralization is locale-specific. Do not force English singular/plural structure onto other locales.
- If a new linted helper or attribute is added, update `.i18n-lint.toml`.

## Red flags

- Hardcoded shipped English text in a linted UI surface.
- Reusing a dictionary key only because the English text matches.
- Adding English text to a non-English locale as filler.
- Splitting one sentence across multiple translation keys.
- Building translated prose by concatenating fragments in the caller.
- Passing raw English strings to `notification/show!`.
- Adding a new UI helper, attribute, alert API, or shipped UI surface without i18n lint coverage.
- Renaming or deleting keys without cleaning affected locale files.
- Function-valued translations that contain logic beyond `str`, `when`, `if`, and `=`.

## Review questions

- Is this text user-facing and inside `.i18n-lint.toml` scope?
- If it is out of scope, should lint coverage be added for the new surface?
- Does an existing `en.edn` key match both semantic owner and textual role?
- Are dynamic values passed as placeholders instead of concatenated into translated text?
- Are rich text, inline links, and complete sentences kept in one translation entry?
- Are locale files cleaned up after key rename/removal?
- Are emoji, icons, punctuation, and pluralization handled in a locale-aware way?

## Required validation when files change

- After changing keys, run `bb lang:validate-translations`.
- After changing UI text, run `bb lang:lint-hardcoded`.
- After editing dictionary files, run `bb lang:format-dicts`.
